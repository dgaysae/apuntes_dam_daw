Antes de ver los componentes de Swing como botones, cajas de texto, etc. es imprescindible comprender **cómo coloca Java estos componentes dentro del contenedor** al invocar el método `add()`. De esto se encarga el **administrador del diseño o _layout manager_**.

Los administradores de diseño son una parte esencial de la creación de interfaces de usuario, ya que determinan las posiciones de los controles en un contenedor. En lenguajes orientados a una sola plataforma, el problema es menor ya que el aspecto es más fácilmente controlable. Pero la filosofía Java está orientada a la portabilidad del código. Por eso este es uno de los apartados más complejos de la creación de interfaces, ya que las medidas y posiciones dependen de la máquina en concreto. 

En otros entornos los componentes se colocan con coordenadas absolutas. En Java se desaconseja esa práctica porque en muchos casos es imposible prever el tamaño de un componente. En su lugar se utilizan administradores de diseño que permiten realizar colocaciones y maquetaciones de forma independiente de las coordenadas. 

El método **`setLayout()`** dentro de la clase `Container` es el encargado de proporcionar un administrador de diseño a un determinado panel. Este método tiene como único parámetro un objeto de tipo `LayoutManager`. `LayoutManager` es una interfaz implementada en diversas clases que sirven para maquetar (`FlowLayout`, `GridLayout`, `BoxLayout`, `BorderLayout`, `GridBagLayout`, ...) 

La adición de elementos se puede hacer del modo habitual, esto es, con el método `add()`. Lo que cambiará al usar un layout manager es el lugar dentro del contenedor en el que se añadirán los componentes. 

> [!warning] Aviso 
> A veces no se muestran los cambios en el contenedor de forma inmediata, y para forzar la visualización hay que utilizar el método **`validate()`** que poseen todos los contenedores. 

**Retomemos el ejemplo del apartado** anterior teniendo esto en cuenta. Como no hemos usado en ningún sitio el método `setLayout()` para asignarle un layout manager a nuestra ventana, esta tomará el `BorderLayout` por defecto, que veremos más adelante.

Vamos a empezar por el más sencillo de todos: el `FlowLayout`.

### 2.3.1. FlowLayout

El administrador de diseño **`FlowLayout`** es el más sencillo. Distribuye los componentes del contenedor **de izquierda a derecha y de arriba abajo**. Es la distribución más fácil y una de las más efectivas, suficiente en la mayoría de los casos. 

> [!abstract] Nota 
> `FlowLayout` distribuye los elementos de la misma forma que escribimos.

**Constructores**: 

* `FlowLayout()`: construye el administrador de tipo flow con alineación centrada y márgenes a 5 píxeles.

* `FlowLayout(int alineación)`: permite indicar la alineación que puede indicarse por alguna de estas constantes de clase: `LEFT`, `RIGHT` o `CENTER`. 

* `FlowLayout(int alineación, int hgap, int vgap)`: permite indicar alineación y la separación horizontal (`hgap`) y vertical (`vgap`) entre los componentes. 

Si aplicamos este layout a nuestro ejemplo:

```java
import javax.swing.*;
import java.awt.*;

public class Main {
    public static void main(String[] args) {
        JFrame frame = new JFrame("Mi primera ventana");
        frame.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        frame.setSize(400, 200);
        frame.setLayout(new FlowLayout());

        JButton bt1 = new JButton("Aceptar");

        frame.add(bt1);

        frame.setVisible(true);
    }
}
```

### 2.3.2. GridLayout

Este administrador de diseño crea distribuciones en forma de malla que posee una serie de columnas y filas. Estas filas y columnas crean celdas de exactamente el mismo tamaño. Los componentes se distribuyen desde la primer celda a la izquierda de la primera fila; y van rellenando fila a fila toda la malla hasta la celda más a la derecha de la última fila. 

Es un layout adecuado para diseñar pantallas complejas que tengan en su interior muchos controles.

En la siguiente captura de pantalla puede verse un panel diseñado con un `GridLayout` de 0 filas y 2 columnas, con separación vertical y horizontal de los componentes.

El método `add()` de los contenedores admite un **segundo parámetro** con el que se puede indicar el **número de la celda** donde se coloca el componente: 

```java
import javax.swing.*;
import java.awt.*;

public class Main {
    public static void main(String[] args) {
        JFrame frame = new JFrame("Mi primera ventana");
        frame.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        frame.setSize(400, 200);
        frame.setLayout(new GridLayout(0, 2));
        
        /*
        Para nuestro ejemplo introducimos la instancia directamente,
        sin variables intermediarias.
        */
        frame.add(new JButton("Button 1"));
        frame.add(new JButton("Button 2"));
        frame.add(new JButton("Button 3"));
        frame.add(new JButton("Long-Named Button 4"));
        frame.add(new JButton("5"));
        
        /*
        El siguiente botón se sitúa en la posición indicada: la 0.
        Es decir, en la primera posición del grid.
        */
        frame.add(new JButton("Odd button"), 0);

        frame.setVisible(true);
    }
}

```

![ud06_swing_GridLayout_01.png](/img/user/attach/Unidad%2006/ud06_swing_GridLayout_01.png)

Aquí, el botón llamado "Odd button" se coloca en la primera casilla (la 0). Si había más casillas ocupadas detrás de ésa, entonces se mueven al siguiente hueco. Y si se añaden más componentes que casillas tenga el contenedor, entonces **se amplía el grid automáticamente**.

### 2.3.3. BorderLayout

Como hemos dicho anteriormente, **si no se le asigna ningún layout al `JFrame`, este toma por defecto el `BorderLayout`**, que organiza la ventana en varias áreas:

![ud06_swing_BorderLayout_01.png](/img/user/attach/Unidad%2006/ud06_swing_BorderLayout_01.png)
> Fuente: https://docs.oracle.com/javase/tutorial/uiswing/layout/border.html

Este layout permite colocar componentes alrededor de los bordes de un contenedor, o en el centro de todos ellos. Por defecto es lo que utiliza AWT (y por lo tanto las clases Swing). Es decir, **es el layout manager que utiliza por defecto** cualquier contenedor si no se le especifica ninguno. Los bordes son `PAGE_START`, `PAGE_END`, `LINE_END`, `LINE_START` y `CENTER`, y se pueden obtener de la misma clase `BorderLayout` ya que son constantes estáticas. 

> [!note] Nota 
> Es posible que veas en varios sitios la equivalencias entre los bordes `PAGE_START` y `NORTH`, `PAGE_END` y `SOUTH`,  `LINE_END` y `EAST` y `LINE_START` y `WEST`. Y aunque no les falta razón, se recomienda usar las primeras, ya que se adaptan a las orientaciones de los distintos idiomas (japonés, árabe, etc.)

En el apartado [[3. Contenedores de Swing]] no especificamos ningún `LayoutManager`, así que por defecto se usaba el `BorderLayout`. Y al añadir un componente (un botón) ocupaba todo el espacio. Esto se debe a que si añadimos elementos sin indicar el border, se añadirá al área central ocupando todo el espacio y solapando a los componentes anteriores:

```java
import javax.swing.*;
import java.awt.*;

public class Main {
    public static void main(String[] args) {
        JFrame frame = new JFrame("Mi primera ventana");
        frame.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        frame.setSize(400, 200);
        frame.setLocationRelativeTo(null);

        JButton bt0 = new JButton("Aceptar 0");
        JButton bt1 = new JButton("Aceptar 1");
        JButton bt2 = new JButton("Aceptar 2");

        frame.add(bt0);
        frame.add(bt1);  // Este solapa al anterior
        frame.add(bt2);  // Este solapa al anterior

        frame.setVisible(true);
    }
}

```

Aunque añadimos los tres botones a la ventana, se van **solapando** uno a otro de forma que solo se muestra el último:

![ud06_swing_BorderLayout_02.png](/img/user/attach/Unidad%2006/ud06_swing_BorderLayout_02.png)

Hay dos formas de utilizar el [BorderLayout](https://docs.oracle.com/javase/tutorial/uiswing/layout/border.html) en el método `add()` del contenedor. Las ilustramos en este ejemplo:

```java
JPanel jp = new JPanel();

// 2 y 3 es el espaciado horizontal y vertical respectivamente:
jp.setLayout(new BorderLayout(2, 3));

jp.add(componente1, BorderLayout.NORTH); 
jp.add("South", componente2);
```

Cualquiera de esas dos formas es válida.

> [!note] Nota 
> El panel central se "come" los paneles adyacentes si detecta que están vacíos. 

> [!abstract] Resumiendo 
> Ahora mismo puede que estés pensando que este layout es el más feo y menos utilizado, pero **observa la ventana en la que estás ahora mismo y su distribución**... Este layout es el más común en cualquier aplicación. En la parte central se pueden incluir muchos componentes, pero debe hacerse en distintos paneles (`JPanel`) y todos ellos contenidos en otro panel, que será el que ocupe toda la parte central. Mientras, en los bordes, podrás encontrar desde una estructura de árbol a la izquierda hasta una columna de propiedades a la derecha, o una barra de estado abajo y una barra de herramientas arriba. 

### 2.3.4. BoxLayout

Permite distribuir componentes **en una fila o columna**. Pensado para filas y columnas de botones, pertenece al paquete `javax.swing` (las anteriores están en `java.awt`). 

Al asignarlo a un contenedor, hay que indicar la dirección del eje:

- **`BoxLayout.X_AXIS`** si queremos hacer una **fila**, es decir, distribuir horizontalmente los componentes (de izquierda a derecha).
- **`BoxLayout.Y_AXIS`** cuando queramos hacer una **columna**, organizándolos verticalmente (de arriba a abajo).

```java
import javax.swing.*;
import java.awt.*;

public class EjemploBoxLayoutVertical {
    public static void main(String[] args) {
        JFrame ventana = new JFrame("BoxLayout Vertical");
        ventana.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        ventana.setSize(400, 200);

        // Creación del panel contenedor
        JPanel panel = new JPanel();
        
        // Se asigna BoxLayout con orientación vertical (Y_AXIS)
        panel.setLayout(new BoxLayout(panel, BoxLayout.Y_AXIS));

        // Añadir componentes
        panel.add(new JButton("Botón 1 - Inicio"));
        panel.add(new JButton("Botón 2 - Centro"));
        panel.add(new JButton("Botón 3 - Fin"));

        ventana.add(panel);
        ventana.setVisible(true);
    }
}
```

Que se vería así:

![ud06_swing_BoxLayout_01.png](/img/user/attach/Unidad%2006/ud06_swing_BoxLayout_01.png)

Al usar un panel en la ventana (`JFrame`) que, por defecto, se organiza con un `BorderLayout`, el panel ocupará todo el espacio del centro.

**Es importante remarcar el uso del panel**. El motivo es que el constructor de `BoxLayout` indica en qué contenedor se va a aplicar.

> [!question] Haz la prueba 🤔
> Prueba a cambiar el código anterior, eliminando el `JPanel` y aplicando el `BoxLayout` directamente en la ventana.
> 
> ¿Qué ocurre...?

Para facilitar su manejo, Swing incluye un contenedor llamado `Box` que está pensado para manipular los componentes insertados en el contenedor. `Box` es una clase que posee diversos métodos estáticos que manipulan internamente el administrador `BoxLayout`. 

En este código se puede apreciar cómo se crea un contenedor `Box`: 

```java
Box horizontal = Box.createHorizontalBox(); 
Boxvertical = Box.createVerticalBox();
```

Después se añaden componentes con el método `add()` de la clase `Box`. Finalmente se coloca el componente `Box` dentro del panel / contenedor deseado. 

Aquí vemos un ejemplo de uso de un `Box`: 

```java
public static void main(String args[]){ 
    // Creamos la ventana e indicamos sus características
    JFrame ventana = new JFrame(); 
    ventana.setLocation(50, 50); 
    ventana.setSize(400, 300); 
    ventana.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
   
    // Creamos los componentes
    JButton boton1 = new JButton("Hola"),
        boton2 = new JButton("Texto largo"), 
        boton3 = new JButton("Hola"),
        boton4 = new JButton("Qué hay");
    
    // Los añadimos a la "caja" (Box)
    Box cajaHorizontal = Box.createHorizontalBox(); 
    cajaHorizontal.add(boton1); 
    cajaHorizontal.add(cajaHorizontal.createHorizontalGlue()); 
    cajaHorizontal.add(boton2); 
    cajaHorizontal.add(cajaHorizontal.createHorizontalGlue()); 
    cajaHorizontal.add(boton3); 
    cajaHorizontal.add(cajaHorizontal.createHorizontalGlue()); 
    cajaHorizontal.add(boton4);
     
    ventana.getContentPane().add(cajaHorizontal); 
    ventana.setVisible(true); 
} 
```

El resultado de la ejecución de este código es: 

![ud06_swing_BoxLayout_02.png](/img/user/attach/Unidad%2006/ud06_swing_BoxLayout_02.png)
> Observa que los botones aparecen separados debido a que hemos usado el método `createHorizontalGlue()`, y la distancia entre componentes se ha ajustado automáticamente.

> [!question]- ¿Qué LayoutManager usa la ventanadel ejemplo anterior? 🤔
> Un **`BorderLayout`**. El contenedor principal de un `JFrame` (llamado `ContentPane`) se crea con el administrador de diseño **`BorderLayout`** por defecto, ya que nunca usamos el `ventana.setLayout(...)`.
> 
> En el ejemplo hemos metido los botones dentro de un contenedor `Box`, al que hemos dicho que use un `BoxLayout` horizontal (`Box.createHorizontalBox()`).
> 
> Al ejecutar `ventana.getContentPane().add(b);`, estamos añadiendo esa `cajaHorizontal` dentro del `BorderLayout` de la ventana. Y como no hemos especificado ninguna zona (`PAGE_START`, `PAGE_END`, etc.), el `BorderLayout` la coloca automáticamente en la zona **`CENTER`**, haciendo que ocupe todo el espacio disponible de la pantalla.

`Box` nos permite usar métodos como el `createHorizontalGlue()` del ejemplo anterior. Los más utilizados son:

1. **`Box.createRigidArea(Dimension)`**: crea un bloque invisible de tamaño fijo, cuando queramos separar componentes por un número exacto de píxeles.
2. **`Box.createHorizontalGlue()` / `Box.createVerticalGlue()`**: actúa como un "pegamento" o resorte elástico,  empujando los componentes hacia los extremos y absorbiendo todo el espacio sobrante.
3. **`Box.createStrut(int)`**: crea un **espacio fijo** de ancho (en horizontal) o alto (en vertical).

Comprueba estos comportamientos jugando con el siguiente ejemplo:

```java
import javax.swing.*;
import java.awt.*;

public class EjemploBoxLayoutAvanzado {
    public static void main(String[] args) {
        JFrame ventana = new JFrame("BoxLayout con Espacios");
        ventana.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        ventana.setSize(400, 300);

        JPanel panel = new JPanel();
        // Distribución horizontal
        panel.setLayout(new BoxLayout(panel, BoxLayout.X_AXIS));

        panel.add(new JButton("Menú"));
        
        // Separación fija de 15 píxeles a la derecha del botón Menú
        panel.add(Box.createRigidArea(new Dimension(15, 0)));
        
        panel.add(new JButton("Editar"));
        
        /*
        Pegamento elástico: empujará lo que venga después hacia
        la derecha de la ventana
        */
        panel.add(Box.createHorizontalGlue());
        
        panel.add(new JButton("Salir"));

        ventana.add(panel);
        ventana.setVisible(true);
    }
}

```


### 2.3.5. GridBagLayout

El **[GridBagLayout](https://docs.oracle.com/javase/tutorial/uiswing/layout/gridbag.html)** es el administrador de diseño **más flexible**, pero también el más complicado de manipular. 

Coloca los componentes en relación a sí mismos y gracias a ello se consigue (con paciencia y algo de suerte) cualquier diseño. Los componentes se distribuyen en un espacio de coordenadas lógicas, que no reales, que son posiciones en el espacio de filas y columnas formadas por el grupo de componentes. De este modo tendremos una cuadrícula imaginaria que se estira formando tamaños basados en los componentes que se acomodan en ella. 

Llegados a este punto, para conseguir un diseño de pantalla verdaderamente profesional, es mucho más cómodo, y hasta imprescindible, **trabajar con programas de diseño WYSIWYG** (como el editor de componentes de NetBeans). 

Esta clase tiene un constructor sin parámetros y métodos para obtener información y configurar la malla de componentes creada. 

#### Posición y propiedades de los componentes

**`GridBagConstraints`** es una clase asociada a este tipo de diseños, y desde ella se controla la posición y propiedades de los componentes añadidos a contenedores `GridBagLayout`. Posee una serie de propiedades como `gridx`, `gridy`, `weightx`, `weighty`, `fill`, etc. Estas propiedades controlan la posición del componente en la malla.

En un `GridBagLayout` no se puede especificar el tamaño de la malla. Ésta se calcula según las necesidades. Esto se basa en lo siguiente: si un componente es añadido, por ejemplo, en la posición `25, 25`; entonces se crea una malla de esas dimensiones. Si el siguiente dice estar en la columna `30`, la malla se amplía para que esto sea posible. 

La colocación de componentes se basa en el método `add` de los contenedores. A este método se le pasa el componente a añadir y el conjunto de restricciones con que queda afectado. Esas restricciones se construyen con la clase `GridBagConstraints`. Las propiedades `gridx` y `gridy` permiten colocar al componente en una celda concreta de la malla. 

Por ejemplo: 

```java
public class VentanaGridBag extends JFrame { 
   GridBagConstraints restricciones = new GridBagConstraints(); 
   public VentanaGridBag (){ 
      getContentPane().setLayout(new GridbagLayout()); 
      añadeGrid(new JButton("Norte"), 1, 0); 
      añadeGrid(new JButton("Sur"), 1, 2); 
      añadeGrid(new JButton("Este"), 2, 1); 
      añadeGrid(new JButton("Oeste"), 0, 1); 
      añadeGrid(new JButton("Centro"), 1, 0); 
   } 
   private void añadeGrid(Component c, int x, int y){ 
      restricciones.gridx = x; 
      restricciones.gridy = y; 
      getContentPane().add(c, restricciones); 
   } 
} 
```

 El resultado sería éste:

![ud06_swing_GridBagLayout_01.png](/img/user/attach/Unidad%2006/ud06_swing_GridBagLayout_01.png)

Se pueden expandir los controles a voluntad cuando no llenan el espacio completo de la celda asignada. Eso se realiza utilizando las propiedades `fill`, `weightx` y `weighty`. La propiedad `fill` permite, por ejemplo, indicar en qué dirección se desplaza el componente para llenar el hueco. 

Además, se pueden utilizar las siguientes constantes estáticas: 

* `GridBagConstraints.BOTH`, el componente se expande en horizontal y en vertical. 

* `GridBagConstraints.HORIZONTAL`, el componente se expande en horizontal. 

* `GridBagConstraints.VERTICAL`, el componente se expande en vertical. 

* `GridBagConstraints.NONE`, el componente no se expande (opción por defecto) 

Así, si, en el ejemplo anterior, tras la línea del setLayout añadimos esto: 

```java
restricciones.fill.GridBagConstraints.BOTH
```

...todos los botones tendrán el mismo tamaño. Y si, además, añadimos esto otro: 

```java
restricciones.weightx = 1.0;   
restricciones.weighty = 1.0;
```

...lograremos expandir las celdas ocupadas hasta llenar la ventana (el valor 0.0 no expande, cualquier valor distinto de 0, sí lo hace).

#### Expansión de celdas

Otro aspecto importante los las propiedades **`gridheight`** y **`gridwidth`**, que permiten **expandir una celda** para que ocupe varias filas o columnas. Como ya habrás supuesto, la propiedad gridheight permite expandir un componente para que ocupe dos filas, y gridwidth permite expandir un componente para que ocupe dos columnas.

Por ejemplo: 

```java
public class VentanaGridBag2 extends JFrame { 
    GridBagConstraints restricciones = new GridBagConstraints();
    
    public VentanaGridBag2() { 
        getContentPane().setLayout(new GridBagLayout()); 
        restricciones.fill = GridBagConstraints.BOTH; 
        restricciones.weightx = 1.0; 
        restricciones.weighty = 1.0; 
        restricciones.gridwidth = 2; 
        añadeGrid(new JButton("Uno"), 0, 0); 
        restricciones.gridwidth = 1; 
        añadeGrid(new JButton("Dos"), 2, 0); 
        restricciones.gridheight = 2; 
        añadeGrid(new JButton("Tres"), 0, 1); 
        restricciones.gridheight = 1; 
        restricciones.gridwidth = 2; 
        añadeGrid(new JButton("Cuatro"), 1, 1); 
        añadeGrid(new JButton("Cinco"), 1, 2); 
        restricciones.gridwidth = 1; 
    }
    
    private void añadeGrid(Component c, int x, int y){ 
        restricciones.gridx = x;
        restricciones.gridy = y; 
        getContentPane().add(c, restricciones); 
    }
} 
```

#### Pesos

Pero sin duda lo más difícil de controlar de `GridBagLayout` son los **pesos**. Los pesos son las propiedades de `GridBagConstraints` **`weigthx`** y **`weighty`**. Su valor es relativo. Es decir, si a un componente le asignamos 0.5 y a otro 1.0, esto significa que el segundo es el doble de grande. El valor 0, significa que se mantendrá como estaba originalmente. 

Los pesos se utilizan para **determinar cómo distribuir el espacio** entre columnas (`weightx`) y filas (`weighty`); esto es importante para especificar el comportamiento durante el redimensionado. A menos que se especifique un valor distinto de cero para weightx o weighty, todos los componentes se situarán juntos en el centro de su contenedor. Esto es así porque cuando el peso es 0,0 (el valor por defecto) el `GidBagLayout` pone todo el espacio extra entre las celdas y los bordes del contenedor.

Generalmente, los pesos se especifican con 0.0 y 1.0 como los extremos, con números entre ellos si son necesarios. Los números mayores indican que la fila o columna del componente deberían obtener más espacio. Para cada columna, su peso está relacionado con el mayor `weightx` especificado para un componente dentro de esa columna (donde cada componente que ocupa varias columnas es dividido de alguna forma entre el número de columnas que ocupa). Lo mismo ocurre con las filas para el mayor valor especificado en `weighty`. 

### 2.3.6. ¿Y trabajar sin layout?

La **ubicación absoluta de componentes** es una forma **desaconsejada** de trabajar, pero lo cierto es que permite una colocación libre y rápida de los elementos ajustándolos al píxel. Consiste en indicar `null` como valor para el método `setLayout()` del contenedor en el que deseamos colocar nuestros componentes. Es decir, no utilizar ningún administrador de diseño (layout):

```java
ventana.setLayout(null);
// o:
ventana.getContentPane().setLayout(null);
```

Con esto estamos diciendo que pondremos las coordenadas de la **posición absoluta** (coordenadas `x` e `y` en el contenedor) y el **tamaño** (ancho y alto) de cada componente antes de añadirlo a la ventana.

> [!note] Nota 
> Si no se desea calcular el tamaño, el método de la clase `Component` `getPreferredSize()` devuelve el tamaño predeterminado que se calculará automáticamente en función de su contenido.

La ubicación absoluta puede utilizarse solo en ventanas pequeñas, que **no vayan a cambiar de tamaño ni tengan demasiados componentes**.

Por ejemplo:

```java
public class pruebaLayoutAbsoluto { 
    public static void main(String args[]){ 
        JButton boton = new JButton("Absoluto"); 
        JFrame ventana = new JFrame(); 
        ventana.setLocation(50, 50); 
        ventana.setSize(400, 300); 
        ventana.setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE); 
        ventana.getContentPane().setLayout(null); 
        boton.setLocation(230, 90); 
        boton.setSize(100, 80); 
        ventana.getContentPane().add(boton); 
        ventana.setVisible(true); 
    } 
}
```

### 2.3.7. Look and feel

Una de las principales mejoras de Swing fue el administrador de apariencia, conocido como **`UIManager`,** para el cual se creó la clase con el mismo nombre (integrada en `javax.swing`). Esta clase permite cambiar la apariencia y la experiencia del usuario (lo que los anglosajones, con su proverbial capacidad de síntesis, llaman *look and feel*) según varios esquemas preestablecidos. 

La idea es que la aplicación Java se integre perfectamente en el entorno en el que se está ejecutando. Es decir, que si la ejecutamos en Windows, *parezca* una aplicación escrita para Windows; pero que, si nos vamos a un Linux, parezca una aplicación Gnome.

Existen varios esquemas predefinidos en `UIManager`, como  Metal (que es el que funciona por defecto), Motif (parecida a la estética del X-Windows de Solaris y otros Unix), Windows (parecida a la estética Windows), Mac (para MacOS) o GTK (para los GNU/Linux basados con escritorio Gnome)

Por ejemplo, si queremos establecer el *look and feel* de Motif (Solaris), usaríamos este código:

```java
try {
    UIManager.setLookAndFeel("com.sun.java.swing.plaf.modif.MotifLookAndFeel");
}
catch (Exception e) { e.printStackTrace(); }
```

Aunque, en general, no querremos darle a nuestra aplicación un aspecto predefinido, sino que adquiera el *look and feel* propio del sistema donde se ejecuta. Para ello, tendremos que detectar el sistema en el que estamos. Esto se hace con el método `getSystemLookAndFeelClassName()`. Lo vemos en este ejemplo:

```java
try {
    UIManager.setLookAndFeel(UIManager.getSystemLookAndFeelClassName());
}
catch (Exception e) { e.printStackTrace(); }
```

En la siguiente tabla mostramos algunos de los L&F existentes:

| Nombre del L&F | Plataforma | Valor devuelto por `getSystemLookAndFeelClassName()` |
| :---: | :---: | :---: |
| `Metal` | `Multi-plataforma` | `javax.swing.plaf.metal.MetalLookAndFeel` |
| `Windows` | `Windows` | `javax.swing.plaf.windows.WindowsLookAndFeel` |
| `Motif` | `Solaris y otros unix con X-Window` | `javax.swing.plaf.motif.MotifLookAndFeel` |
| `Mac` | `MacOS` | `javax.swing.plaf.mac.MacLookAndFeel` |
| `GTK` | `GNU/Linux` | `javax.swing.plaf.gtk.GTKLookAndFeel` |
