Es posible que te hayas fijado en que, en algunos proyectos complejos (como los que usan Maven para integrar JavaFX, por ejemplo), aparece un archivo **`module-info.java`** en el directorio raíz del proyecto. A veces incluso hemos tenido que tocarlo un poco sin saber muy bien qué estábamos haciendo.

¿Qué es `module-info.java` y qué son los módulos de Java? Vamos a tratar de responderlo en esta sección.

## 8.1. Qué es `module-info.java` y por qué existe

Hasta Java 8, todo en Java funcionaba con `CLASSPATH`, que es básicamente una lista gigante de directorios y archivos donde Java busca las clases con las que compilar nuestros programas. 

Esto funciona, pero tenía problemas importantes en proyectos grandes:

* Todo era visible para todos, incluso lo que no quieres exponer.

* Era muy fácil tener conflictos de versiones entre librerías o de nombres de clases repetidos.

* Algunos errores de dependencia aparecen solo en tiempo de ejecución, no al compilar, lo que los hace especialmente difíciles de depurar.

Para resolver esto, en Java 9 se introdujo el sistema de módulos llamado **Project Jigsaw**. La idea es que un **módulo sea una unidad autocontenida de código**, con:

* Un nombre único

* Dependencias explícitas hacia otros módulos

* Control sobre qué paquetes expone al exterior

El archivo **`module-info.java`** es donde se define todo esto. Es una especie de “contrato” que tu módulo, es decir, lo que tú estás programando, firma con el resto del ecosistema Java: dice qué módulos necesita, qué módulos ofrece  y qué partes de sí mismo abre a otros módulos.

## 8.2. Anatomía de `module-info.java`

Para entenderlo mejor, vamos a trabajar con **un module-info.java imaginario pero totalmente realista**. Imagina que estamos programando un módulo (una sección) de una tienda online. Nuestro module-info.java podría tener este aspecto:

```java
module com.tienda.core {
    requires java.sql;
    requires transitive com.tienda.api;

    exports com.tienda.core.servicios;

    opens com.tienda.core.entidades to org.hibernate.orm.core;
}
```

Vamos a entenderlo línea por línea:

1. **`module com.tienda.core`**  
   Aquí se define el módulo en el que estamos trabajando. Es obligatorio y debe tener un nombre único. Es el identificador de tu módulo ante Java y ante otros módulos.

2. **`requires java.sql`**  
   Esta línea dice: “este módulo necesita el módulo `java.sql` para funcionar”. Es similar a importar una librería en Maven o Gradle, pero mucho más formal. La diferencia clave es que ahora las dependencias son parte del contrato: si alguien intenta usar tu módulo sin estas dependencias, Java se quejará durante la compilación.

3. **`requires transitive com.tienda.api`**  
   “transitive” significa que cualquier módulo que dependa de tu módulo automáticamente también necesitará `com.tienda.api`. Es como decir: “si alguien usa mi módulo, también tendrá acceso a esta dependencia”.

   Esto es muy útil para APIs o librerías donde ciertas dependencias son parte de tu interfaz pública.

4. **`exports com.tienda.core.servicios`**  
   Esto define qué paquetes de tu módulo son visibles desde otros módulos. La regla que muchos encuentran chocante al principio es:

   Incluso si una clase es `public`, si su paquete no está exportado, otros módulos no la verán.

   Es decir, `public` ya no significa “todo el mundo puede usar esto”, sino “visible dentro del módulo y si exportas el paquete, también afuera”. Esto te da encapsulación fuerte que antes no existía en Java.

5. **`opens com.tienda.core.entidades to org.hibernate.orm.core`**
   Esto es un caso especial para usar el **mecanismo de reflexión** que hemos visto antes. 

   Muchos frameworks, como Hibernate o Jackson, necesitan inspeccionar clases en tiempo de ejecución. Con módulos, la reflexión no puede acceder a paquetes privados a menos que abras específicamente esos paquetes.

   `opens` abre el paquete a todos los módulos para reflexión.

   `opens ... to ...` abre solo a los módulos que tú especifiques.

   Esto evita problemas donde frameworks dejaban de funcionar de repente porque Java decidió que tu código “no era visible”.

## 8.3. Pero ¿tengo que usar siempre `module-info.java`?

**Absolutamente NO.**

Con el sistema de módulos puedes controlar exactamente qué parte de tu código se ve y qué se oculta. Esto es **un paso importante hacia el diseño de código más seguro y mantenible**. Java se asegura de que tu módulo tenga todas las dependencias que necesita y falla en tiempo de compilación si no es así.

Sin embargo, si estás programando un sistema pequeño, una aplicación simple, un prototipo, etc, el sistema de módulos solo será un engorro que no te ayudará en nada.

Muchos proyectos profesionales aún no usan módulos, incluso con Java 17 o 21, pero entenderlos te da ventaja en arquitectura y diseño.

Cuando uses un arquetipo para crear un proyecto (por ejemplo, cuando creas un proyecto Maven JavaFX con VS Code), el propio arquetipo, salvo que sea muy, muy antiguo, incluirá el `module-info.java`, así que no tendrás que hacer nada particular… excepto si necesitas añadir manualmente alguna dependencia.
