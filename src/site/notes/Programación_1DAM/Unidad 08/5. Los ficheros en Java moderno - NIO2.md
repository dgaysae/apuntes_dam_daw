## 5.1. ¿En qué sentido es mejor NIO2 que `java.io`?

Durante muchos años, el manejo de ficheros en Java se realizó utilizando las clases del paquete **java.io**. Este conjunto de clases forma parte del lenguaje desde sus primeras versiones y, como hemos visto, permite realizar operaciones básicas como leer y escribir ficheros, copiar datos o trabajar con flujos de entrada y salida.

Sin embargo, con el paso del tiempo sacó a la luz **algunas limitaciones** de esta API, como:

* Algunas operaciones habituales resultan más complicadas de lo necesario

* Algunas tareas requieren bastante código repetitivo

* Ciertas funcionalidades modernas de los sistemas de ficheros no están bien integradas

Para solucionar estas carencias, Java introdujo una nueva API de entrada y salida llamada **NIO** (**New Input/Output**). Más adelante, a partir de Java 7, esta API se amplió considerablemente con una versión más completa y potente conocida como **NIO2**.

NIO2 incorpora **nuevas clases y mecanismos** diseñados para trabajar con ficheros y directorios de una forma más sencilla, más flexible y más cercana a las capacidades reales de los sistemas operativos actuales. Entre otras cosas, permite gestionar rutas de fichero de forma más clara, realizar operaciones comunes con menos código y acceder a información adicional sobre los ficheros.

Esto **no significa que las clases del paquete java.io hayan desaparecido**. De hecho, siguen existiendo y continúan utilizándose en muchos programas y bibliotecas. Pero en proyectos modernos es habitual preferir las herramientas de NIO2 y recurrir a las clases más antiguas solo cuando resulta necesario.

En los apartados siguientes veremos las clases más importantes de esta API moderna y cómo utilizarlas para realizar las operaciones más habituales con ficheros y directorios.

## 5.2. Path y rutas de fichero

Para trabajar con ficheros, es imprescindible poder identificar dónde se encuentra un fichero dentro del sistema de ficheros. Esto se hace, como sabemos, mediante su **ruta** (o **path**).

En la API clásica de Java, esta tarea se realizaba principalmente con la clase File. Sin embargo, NIO2 introduce una nueva forma de representar rutas mediante la **interfaz Path**, que forma parte del paquete java.nio.file.

La interfaz Path representa la ubicación de un fichero o directorio en el sistema de ficheros, pero *no el fichero en sí*. Es importante entender esta diferencia: un objeto Path simplemente **describe una ruta**, es decir, una especie de “dirección”. El fichero puede existir o no; el objeto Path solo indica **dónde debería estar**.

### Creación de objetos Path

La forma más habitual de crear un objeto Path es mediante el método estático **get()** de la clase Paths. Este método recibe como argumento la ruta del fichero y devuelve un objeto Path que la representa.

`Path ruta = Paths.get("datos.txt");`

En este caso se crea un objeto Path que representa un fichero llamado datos.txt situado en el directorio de trabajo actual del programa.

También es posible especificar rutas más completas, incluyendo directorios:

`Path ruta = Paths.get("datos/clientes.txt");`

Otra posibilidad es construir la ruta a partir de varios elementos separados. Esto permite que Java se encargue automáticamente de usar el separador de directorios adecuado para cada sistema operativo.

`Path ruta = Paths.get("datos", "clientes", "clientes.txt");`

Este enfoque es especialmente útil cuando se quiere escribir código que funcione correctamente tanto en Windows como en Linux o macOS.

### Algunas operaciones básicas con Path

La interfaz `Path` ofrece numerosos métodos para trabajar con rutas y manipularlas. Estos métodos permiten obtener información sobre la ruta o construir nuevas rutas a partir de otras existentes.

Algunos de los métodos más útiles de `Path` son:

* **`getFileName()`**: devuelve el nombre del fichero o último elemento de la ruta

* **`getParent()`**: devuelve la ruta padre

* **`getRoot()`**: devuelve la raíz de la ruta

* **`getNameCount()`**: devuelve el número de elementos de una ruta

* **`toAbsolutePath()`**: convierte la ruta en absoluta

* **`equals(Object otroPath)`**: compara si dos rutas son iguales

Aquí tienes un pequeño ejemplo de uso de algunos de ellos:

```java
Path ruta = Paths.get("clientes.txt");

// Imprime nombre del archivo
System.out.println(ruta.getFileName());

// Imprime el directorio del archivo
System.out.println(ruta.getParent());

// Imprime la ruta absoluta del archivo
System.out.println(ruta.toAbsolutePath());
```

### Path como punto de partida

En NIO2, los objetos Path suelen actuar como **punto de partida para casi todas las operaciones con ficheros**. Una vez que se dispone de un objeto Path, puede utilizarse junto con otros métodos de la clase Files para realizar tareas como comprobar si un fichero existe, leer o escribir contenido, copiar o mover ficheros o recorrer directorios.

En los siguientes apartados veremos cómo la clase `Files` utiliza los objetos `Path` para realizar estas operaciones de forma sencilla y directa. De este modo, `Path` se convierte en una pieza fundamental de NIO2.

## 5.3.  Operaciones a nivel de fichero con la clase Files 

Una vez que tenemos claro cómo representar la ubicación de un fichero con Path, el siguiente paso en NIO2 es aprender a **realizar operaciones sobre ese fichero o directorio**. Para ello, Java nos ofrece la **clase Files**, que forma parte del paquete java.nio.file. *¡No confundir con la clase File del java.io clásico!*

Files permite realizar prácticamente todas las operaciones habituales que necesitamos, de manera directa, sin tener que lidiar con los detalles de bajo nivel que antes hacíamos con java.io.

Casi todos los métodos de Files son estáticos, por lo que no necesitamos crear un objeto especial para usarlo. Solo basta con pasarle un Path y la operación se realiza de forma clara y concisa.

### Comprobar la existencia y el tipo de un fichero

Antes de trabajar con un fichero, casi siempre necesitamos saber si existe y qué tipo de objeto representa: ¿es un fichero normal, un directorio, o quizá no existe?

La clase Files nos ofrece métodos muy directos para estas comprobaciones, como:

* **`exists()`**: devuelve true si un fichero o directorio existe

* **`isDirectory()`**: devuelve true si un `Path` es un directorio

* **`isRegularFile()`**: devuelve true si un `Path` es un fichero.

Los vemos en acción en este ejemplo:

```java
Path ruta = Paths.get("datos/clientes.txt");

// ¿Existe el archivo?
if (Files.exists(ruta)) {
    System.out.println("El archivo existe");
}

// ¿Es un directorio?
if (Files.isDirectory(ruta)) {
    System.out.println("Es un directorio");
}

// ¿Es un archivo normal?
if (Files.isRegularFile(ruta)) {
    System.out.println("Es un archivo normal");
}
```

### Crear y borrar ficheros y directorios

`Files` tiene métodos para crear y borrar tanto ficheros como carpetas:

* **`createFile()`**: crea un fichero

* **`createDirectory()`**: crea un directorio

* **`delete()`**: borra un fichero o directorio.

He aquí un pequeño ejemplo:

```java
// Crear un archivo nuevo
Files.createFile(Paths.get("datos/nuevo.txt"));

// Crear un directorio
Files.createDirectory(Paths.get("datos/clientes"));

// Borrar un archivo o directorio
Files.delete(Paths.get("datos/nuevo.txt"));
```

Además, hay variantes que no lanzan excepción si el fichero ya existe, lo que facilita manejar situaciones comunes sin tener que escribir mucho código adicional.

### Copiar y mover ficheros

Otra operación habitual es copiar o mover ficheros. Con `Files` esto se hace de forma directa y muy legible:

* **`copy()`**: copia un `Path` de origen en otro de destino
* **`move()`**: mueve un `Path` de origen a otro de destino

Como siempre, aquí tienes un ejemplo de uso:

```java
Path origen = Paths.get("datos/clientes.txt");
Path destino = Paths.get("backup/clientes.txt");

// Copiar archivo
Files.copy(origen, destino, StandardCopyOption.REPLACE_EXISTING);

// Mover archivo
Files.move(origen, destino, StandardCopyOption.REPLACE_EXISTING);
```


El parámetro **`StandardCopyOption.REPLACE_EXISTING`** indica que, si el fichero de destino ya existe, será reemplazado. Esto evita los errores típicos de “fichero ya existe”.

### Acceso a metadatos

`Files` no solo lee y escribe texto, sino que también puede obtener información sobre el propio fichero, como su tamaño, tipo o permisos (estos son los “metadatos” del fichero):

* **`size()`**: tamaño del fichero
* **`getLastModifiedTime()`**: fecha última modificación
* **`getOwner()`**: propietario del fichero
* **`isReadeable()`**: ¿Tiene permisos de lectura?
* **`isWriteable()`**: ¿Tiene permisos de escritura?
* **`isExecutable()`**: ¿Tiene permisos de ejecución?
* **`isExecutable()`**: ¿Es un fichero o directorio oculto?

Aquí tienes un pequeño ejemplo de uso:

```java
Path ruta = Paths.get("datos/clientes.txt");

// Tamaño del archivo en bytes
long tamaño = Files.size(ruta);

// Última fecha de modificación
FileTime ultimaModificacion = Files.getLastModifiedTime(ruta);

// Permisos y atributos
boolean esLegible = Files.isReadable(ruta);
boolean esEscribible = Files.isWritable(ruta);
```

Esto convierte a Files en la herramienta estrella de NIO2 y nos permite trabajar con ficheros de forma mucho más segura, moderna y legible que con la API clásica de java.io.

## 5.4.  Operaciones a nivel de contenido del fichero con la clase Files

Una de las tareas más habituales cuando trabajamos con ficheros es, por supuesto, leer su contenido y escribir nuevos datos. NIO2 nos facilita esto de **manera más sencilla que la API clásica** de `java.io`.

En NIO2, la lectura y escritura de ficheros se realiza principalmente a través de objetos `Path` junto con los métodos de la clase `Files`. La combinación de estas dos poderosas clases nos permite escribir código más claro, seguro y menos propenso a errores.

### Leer y escribir texto

La clase `Files` sirve para todo; también nos permite, de manera muy simple, escribir y leer texto en un fichero con los métodos:

* **`write()`**: envía texto a un Path. Puede ser un `String` o un `List<String>`, da igual.

* **`readAllLines()`**: lee todo el contenido de un `Path` y lo devuelve en un `List<String>`

* **`readString()`**: lee todo el contenido de un `Path` y lo devuelve en un único `String`

Aquí tienes un ejemplo sencillo de uso:

```java
// Escribimos varias líneas en un archivo
List<String> lineas = List.of("Juan", "Marta", "Luis");
Files.write(Paths.get("datos/nombres.txt"), lineas);

// Leemos todas las líneas de un archivo
List<String> contenido = Files.readAllLines(Paths.get("datos/nombres.txt"));

// Mostramos las líneas leídas por pantalla
for(String linea: contenido) {
    System.out.println(linea);
}
```

En comparación con java.io, esto evita tener que abrir y cerrar los flujos manualmente, lo que hace que el código sea más corto, más claro y menos propenso a errores.

`Files` tiene más métodos, además de los tres que hemos mencionado más arriba, que permiten manipular los ficheros de manera más avanzada. Por ejemplo, `Files.lines()` nos devuelve el fichero convertido en un Stream, que podemos procesar con expresiones lambda, dando lugar a código ultracompacto como este:

```java
try (Stream<String> stream = Files.lines(ruta)) {
    stream.filter(linea -> linea.startsWith("A"))
          .forEach(System.out::println);
}
```

### Leer y escribir bytes

La clase Files, por supuesto, también permite trabajar con ficheros binarios mediante flujos de bytes, como FileInputStream y FileOutputStream.

Los métodos más útiles para ello son:

* **`write()`**: escribe un array de bytes en un fichero. Se puede usar para cualquier tipo de datos binarios, como imágenes, audio o ficheros .bin.

* **`readAllBytes()`**: lee todo el contenido de un fichero y lo devuelve como un array de bytes.

* **`newInputStream()` / `newOutputStream()`**: permiten leer o escribir bytes de forma secuencial, útil cuando el fichero es muy grande y no queremos cargarlo todo en memoria de golpe.

En este ejemplo puedes ver cómo se escriben varios bytes en un fichero y luego se recuperan:

```java
// Creamos un array de bytes
byte[] datos = {0x01, 0x02, 0x03, 0x04};  // Números binarios

// Escribimos los bytes en un archivo
Files.write(Path.of("datos/archivo.bin"), datos);

// Recuperamos los bytes del archivo
byte[] contenido = Files.readAllBytes(Path.of("datos/archivo.bin"));

// Mostramos los bytes leídos por pantalla
for (byte b : contenido) {
     System.out.println(b);
}
```

En comparación con java.io, usar Files para leer y escribir bytes evita tener que abrir y cerrar manualmente InputStream o OutputStream en muchos casos, lo que hace que el código sea más corto, más claro y menos propenso a errores.

### **Serialización de objetos** {#serialización-de-objetos-1}

NIO2 puede serializar objetos igual que `java.io`, es decir, con **`ObjectInputStream`** y **`ObjectOutputStream`**, solo que, en este caso, esas clases trabajarán con sobre Files. 

Basta con crear un `InputStream` o un `OutputStream` a partir de `Files` y usarlo para construir el `ObjectInputStream` o el `ObjectOutputStream` con los métodos **`newInputStream()`** y **`newOutputStream()`**.

Como es lógico, el objeto que se pretende serializar debe implementar el interface `Serializable`, como hacíamos con java.io (repasa el apartado correspondiente si no recuerdas cómo se hace esto).

Aquí tienes un pequeño ejemplo en el que se escribe un objeto de tipo `Persona` en un fichero y luego se recupera.

```java
Persona p = new Persona("Juan");

// Serializamos con ObjectOutputStream
try (ObjectOutputStream oos = new ObjectOutputStream(
    Files.newOutputStream(Path.of("datos/persona.dat")))) {
        oos.writeObject(p);
}

// Leemos con ObjectInputStream
try (ObjectInputStream ois = new ObjectInputStream(
    Files.newInputStream(Path.of("datos/persona.dat")))) {
        Persona p2 = (Persona) ois.readObject();
        System.out.println(p2.nombre);
}
```

### Leer y escribir con buffer: BufferedReader y BufferedWriter con NIO2

Otra forma de leer y escribir ficheros de texto con NIO2 es mediante **buffers de lectura y escritura**. 

Esto nos da más control sobre el proceso, especialmente si necesitamos manipular los datos mientras se leen o escriben, y hace que las operaciones de entrada/salida sean mucho más eficientes.

Para lectura y escritura con buffer, Files dispone de los métodos **`newBufferedReader()`** y **`newBufferedWriter()`**. Mira cómo se usan:

```java
// Escritura con buffer
try (BufferedWriter writer = Files.newBufferedWriter(ruta)) {
    writer.write("Pedro");
    writer.newLine();
    writer.write("Lucía");
}
// Lectura con buffer
try (BufferedReader reader = Files.newBufferedReader(ruta)) {
    String linea;
    while ((linea = reader.readLine()) != null) {
        System.out.println(linea);
    }
}
```

Los buffers son especialmente útiles para ficheros grandes, ya que reducen la cantidad de accesos al disco, haciendo la operación más eficiente.

## 5.5.  Manejo de directorios con la clase `Files`

Cuando trabajamos con ficheros, no solo necesitamos leer y escribir su contenido: en muchas ocasiones también necesitamos organizar, explorar o modificar las carpetas que los contienen.

Con NIO2, el manejo de directorios se hace también con una combinación de Path y Files. Te lo muestro con algunos ejemplos rápidos:

Crear un directorio:

```java
Path directorio = Paths.get("datos/clientes");
Files.createDirectory(directorio);
```

Crear varios directorios (carpeta dentro de carpeta):

```java
Path directorioCompleto = Paths.get("datos/clientes/2026");
Files.createDirectories(directorioCompleto);
```

Borrar directorio (debe de estar vacío):

```java
Files.delete(directorio); // solo funciona si la carpeta está vacía
```

Listar el contenido de un directorio (como nos devuelve un Stream, lo procesamos con una lambda):

```java
try (Stream<Path> contenidos = Files.list(Paths.get("datos"))) {
    contenidos.forEach(System.out::println);
}
```

Comprobar existencia y tipo de un directorio

```java
Path ruta = Paths.get("datos/clientes");

if (Files.exists(ruta)) {
    if (Files.isDirectory(ruta)) {
        System.out.println("Es un directorio");
    } else if (Files.isRegularFile(ruta)) {
        System.out.println("Es un archivo normal");
    }
}
```

## 5.6.  Acceso aleatorio directo: `FileChannel` + `ByteBuffer`

Las operaciones de lectura y escritura que hemos visto son secuenciales, que es la forma habitual de acceder a los ficheros pues, como ya hemos discutido, es más sencillas y suficiente la mayoría de las veces.

Si necesitamos hacer **acceso aleatorio directo**, es decir, leer o escribir en una posición concreta del fichero, con NIO2 puedes hacerlo con la clase **`FileChannel`**. Aunque hay otros medios, este es el más flexible… aunque al principio resulta un poco engorroso.

### La clase `FileChannel`

`FileChannel` se construye, como `Files`, a partir de un `Path`, y nos proporciona un montón de métodos para acceso aleatorio directo.

1) **Métodos para abrir el canal:**  
  * **open(Path path, OpenOption options…)** → devuelve un FileChannel abierto con una o varias de estas opciones, separadas por comas:   
    * StandardOptions.CREATE → crea el fichero (siempre, aunque exista)  
    * StandardOptions.CREATE_NEW → crea el fichero si no existe (falla si existe)  
    * StandardOptions.READ → habilita la lectura  
    * StandardOptions.WRITE → habilita la escritura  
    * StandardOptions.APPEND → todos los write() se añaden al final del fichero  
    * StandardOptions.DELETE_ON_CLOSE → borra el fichero nada más cerrarlo  
1) **Métodos de posicionamiento:**  
  * **position()** → devuelve la posición actual del canal.  
  * **position(long newPosition)** → establece la posición del canal para la próxima lectura/escritura.  
  * **size()** → devuelve el tamaño actual del fichero (en bytes).  
  * **truncate(long size)** → reduce el tamaño del fichero a la cantidad de bytes indicada.  
1) **Métodos para cerrar el canal:**  
  * **force(boolean metaData)** → fuerza que los cambios se escriban en disco; metaData = true hace que se incluyan los metadatos que se hayan podido modificar.  
  * **close()** → cierra el canal.  
1) **Métodos de lectura/escritura:**  
  * **read(ByteBuffer dst)** → lee desde la posición actual.  
  * **read(ByteBuffer dst, long position)** → lee desde una posición específica.  
  * **write(ByteBuffer src)** → escribe desde el buffer a la posición actual.  
  * **write(ByteBuffer src, long position)** → escribe en una posición específica.

Además, `FileChannel` tiene varios métodos de **mapeo**, que se usan para transferir porciones del fichero a la memoria para trabajar más rápidamente con ellos. No entraremos en ellos en esta introducción, pero no son demasiado difíciles de manejar y los necesitarás si tus ficheros son muy grandes y haces un uso masivo de ellos en algún programa.

Si observas la lista de métodos para lectura y escritura, verás que usan la clase **`ByteBuffer`**, otra clase de NIO2 que al principio resulta un poco liosa, así que vamos a verla con cierta calma.

### Cómo funciona un `ByteBuffer`

Un `ByteBuffer` funciona como “bandeja de entrada” o “bandeja de salida” de los datos. Internamente mantiene tres números que especifican su estado:

* **`position`**: dónde estás escribiendo o leyendo el buffer

* **`capacity`**: cuánto espacio total tienes. Lo defines al crear el buffer.

* **`limit`**: hasta dónde puedes leer (límite de datos efectivos del buffer; no tiene por qué coincidir con la capacidad, pues el buffer puede no haberse llenado de datos completamente)

El `ByteBuffer`, además, tiene **dos modos** de funcionamiento: **escritura o lectura**. Se permuta de uno a otro con el método **`flip()`**, como si fuera un interruptor.

Cuando se crea, el `ByteBuffer` siempre está en “modo escritura”, como una libreta en blanco esperando para llenarse. Al ir metiendo cosas en el buffer con el método **`put()`**, él va avanzando automáticamente la posición de escritura mientras acumula datos en su interior.

En cuanto haces `flip()`, el `ByteBuffer` pasa de “modo escritura” o “modo lectura”, y puedes empezar a sacar datos de él. Si intentas leer antes de hacer `flip()`, el buffer te dirá que no hay datos útiles para leer.

(Lo que hace internamente `flip()` es colocar el limit donde estaba la position, y resetear la position a 0. Es decir, convierte “lo que se ha escrito” en “lo que se puede leer”).

A partir de ahí, entras en modo lectura y puede hacer **`get()`**. El buffer va avanzando la posición igual que antes, pero ahora consumiendo datos en lugar de acumularlos.

Cuando terminas de leer, lo más común es hacer **`clear()`** para limpiar completamente el buffer.

Aquí tienes un ejemplo completo de uso de FileChannel + `ByteBuffer`:

```java
Path path = Paths.get("datos.bin");

try (FileChannel fc = FileChannel.open(
        path,
        StandardOpenOption.CREATE,
        StandardOpenOption.READ,
        StandardOpenOption.WRITE)) {
    
    // -------- ESCRIBIR --------
    // Creamos un buffer de 4 bytes de capacidad (cabe un int)
    ByteBuffer buffer = ByteBuffer.allocate(4);
    buffer.putInt(12345);  // Metemos un int en el buffer
    buffer.flip();         // Lo ponemos en “modo lectura”
    
    long posicion = 20;    // Escribiremos en la posición 20 del archivo
    fc.write(buffer, posicion);

    // -------- LEER --------
    // De nuevo, creamos un buffer de 4 bytes (para un int)
    ByteBuffer bufferLectura = ByteBuffer.allocate(4);

    fc.read(bufferLectura, posicion);  // Leemos en la posición 20 del fichero

    // Ponemos el buffer en “modo lectura”
    bufferLectura.flip();
    int valor = bufferLectura.getInt();  // Recuperamos info del buffer

    System.out.println(valor);
}
```
