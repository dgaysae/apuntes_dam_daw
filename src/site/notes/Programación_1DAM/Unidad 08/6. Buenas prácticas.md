Trabajar con ficheros en Java puede parecer sencillo al principio: abrir, leer, escribir y cerrar. Sin embargo, en cuanto los proyectos crecen, surgen **problemas sutiles** que pueden causar errores difíciles de detectar, como ficheros bloqueados, pérdidas de datos, fugas de memoria o problemas de codificación de caracteres.

Esta sección reunimos un conjunto de buenas prácticas modernas que nos permiten:

* Manejar ficheros y directorios de manera segura y confiable.  
* Evitar errores comunes al leer y escribir datos.  
* Escribir código claro, limpio y fácil de mantener, tanto con NIO2 y como con la API clásica ([java.io](http://java.io)).


## 6.1. try-with-resources

Uno de los problemas más comunes al trabajar con ficheros en Java —ya sea con la API clásica (`java.io`) o con NIO2— es **olvidar cerrar los recursos** después de usarlos. 

Un recurso puede ser un fichero, un flujo de lectura/escritura o cualquier objeto que consuma recursos del sistema, como una conexión con una base de datos. ¡Si no los cerramos correctamente, podemos encontrarnos con ficheros bloqueados, fugas de memoria o comportamientos imprevisibles en nuestra aplicación!

Para resolver esto de forma elegante y segura, Java introdujo el mecanismo llamado **`try-with-resources`**, disponible desde Java 7. Este mecanismo permite **abrir recursos dentro de un bloque `try`** y garantiza que se cierren automáticamente al finalizar el bloque, incluso si ocurre una excepción durante la operación.

La sintaxis de `try-with-resources` es muy sencilla. Supongamos que queremos leer un fichero con `BufferedReader` y `Files`:

```java
Path ruta = Paths.get("datos/clientes.txt");

try (BufferedReader reader = Files.newBufferedReader(ruta)) {
    String linea;
    while ((linea = reader.readLine()) != null) {
        System.out.println(linea);
    }
} catch (IOException e) {
    System.err.println("Error al leer el archivo: " + e.getMessage());
}
```

En este ejemplo:

* El recurso `reader` **se abre dentro de los paréntesis del `try`**.  
* Cuando el bloque `try` termina, el recurso **se cierra automáticamente**, incluso si ocurre una excepción dentro del bloque.  
* **¡No necesitamos llamar a `reader.close()` manualmente!** Esto reduce errores y hace el código más limpio.

Se pueden abrir varios recursos a la vez a la vez en un bloque `try`, separándolos con punto y coma. Por ejemplo:

```java
try (BufferedReader reader = Files.newBufferedReader(Paths.get("clientes1.txt"));
     BufferedWriter writer = Files.newBufferedWriter(
               Paths.get("clientes2.txt"))
    ) {
   ...etc...
```

## 6.2. Manejo de excepciones de E/S

**Trabajar con ficheros siempre implica incertidumbre**. Los potenciales errores de ejecución son casi infinitos: el fichero puede no existir, estar bloqueado por otro proceso, el disco puede estar lleno o puede haber problemas de permisos.

Por eso, en todas las operaciones de entrada/salida pueden lanzar excepciones. Aprender a manejarlas correctamente es clave para escribir programas robustos y evitar fallos inesperados.

Las excepciones más comunes al trabajar con ficheros (tanto en NIO2 como en java.io) son:

* **`IOException`**: la más general. Indica que ha ocurrido un problema de entrada/salida, como fallo al leer o escribir un fichero.

* **`FileNotFoundException`**: se lanza cuando intentamos acceder a un fichero que no existe (más común en java.io).

* **`AccessDeniedException`**: ocurre cuando el programa no tiene permisos para leer, escribir o borrar un fichero o directorio.

* **`NoSuchFileException`**: cuando el fichero o directorio que queremos manipular no existe (NIO2).

* **`FileAlreadyExistsException`**: cuando intentamos crear un fichero o directorio que ya existe y no hemos indicado que se sobrescriba.

En cuanto a las mejores estrategias para manejar las excepciones de E/S, podemos mencionar las siguientes:

1. **Capturar primero las excepciones específicas.** Siempre es recomendable capturar primero las excepciones más específicas y luego las más generales:
  
    ```java
    Path ruta = Paths.get("datos/clientes.txt");
    try {
        List<String> lineas = Files.readAllLines(ruta);
        lineas.forEach(System.out::println);
    } catch (NoSuchFileException e) {
        System.err.println("El archivo no existe: " + e.getFile());
    } catch (AccessDeniedException e) {
        System.err.println("No se tiene permiso para acceder al archivo: "
            + e.getFile());
    } catch (IOException e) {
        System.err.println("Error de E/S: " + e.getMessage());
    }
    ```
  
  Capturar excepciones específicas en primer lugar nos permite dar mensajes claros al usuario o tomar decisiones distintas según el error.

1. **Usar `try-with-resources`**. Ya hemos mencionado antes que el código queda más claro y se garantiza que los recursos se cierran. ¡Todo son ventajas!

2. **Propagar excepciones.** En algunos casos no es conveniente manejar la excepción en el mismo método; podemos propagarla al método que llama, usando throws. Esto es útil en **aplicaciones grandes**, donde el manejo de errores se centraliza en un nivel superior de la aplicación.

    ```java
    public static void leerArchivo(Path ruta) throws IOException {
        List<String> lineas = Files.readAllLines(ruta);
        lineas.forEach(System.out::println);
    }
    ```

4. **Ofrecer mensajes claros y precisos al usuario** cuando ocurre un error. El típico `e.printStackTrace()` está bien durante el desarrollo, pero no es útil para el usuario final. Y siempre hay que procurar que el programa se recupere del error y continúe ejecutándose.

## 6.3. Codificación de caracteres sin problemas

Cuando trabajamos con **ficheros de texto**, uno de los **problemas más frecuentes y difíciles de detectar** proviene de la **codificación de caracteres**. 

Si el programa lee o escribe un fichero usando una codificación distinta a la que fue guardado, nos encontraremos con caracteres extraños, símbolos raros o errores de lectura. Por eso, la recomendación actual es **usar UTF-8** siempre que sea posible.

Con NIO2 es fácil el trabajo con codificaciones. Al crear un `BufferedReader` o `BufferedWriter` a partir de un `Path`, podemos especificar explícitamente la codificación:

```java
Path ruta = Paths.get("datos/clientes.txt");

// Lectura en UTF-8
try (BufferedReader reader = Files.newBufferedReader(ruta, StandardCharsets.UTF_8)) {
    String linea;
    while ((linea = reader.readLine()) != null) {
        System.out.println(linea);
    }
} catch (IOException e) {
    System.err.println("Error leyendo el archivo: "
                        + e.getMessage());
}

// Escritura en UTF-8
try (BufferedWriter writer = Files.newBufferedWriter(ruta,
                                       StandardCharsets.UTF_8)) {
    writer.write("José\n");
    writer.write("Marta\n");
} catch (IOException e) {
    System.err.println("Error escribiendo el archivo: "
                        + e.getMessage());
}
```

Con java.io, se puede hacer algo parecido si usamos la secuencia **`InputStreamReader`** → **`FileInputStream`** → **`BufferedReader`** para leer y la secuencia  **`OutputStreamReader`** → **`FileOutputStream`** → **`BufferedWriter`** para escribir, como en este ejemplo. Es un poco más engorroso que con NIO2, pero funciona:

```java
// Lectura en UTF-8
try (BufferedReader reader = new BufferedReader(
         new InputStreamReader(new FileInputStream("datos/clientes.txt"),
                                   StandardCharsets.UTF_8))) {
    String linea;
    while ((linea = reader.readLine()) != null) {
        System.out.println(linea);
    }
}

// Escritura en UTF-8
try (BufferedWriter writer = new BufferedWriter(
         new OutputStreamWriter(new FileOutputStream("datos/clientes.txt"),
                                    StandardCharsets.UTF_8))) {
    writer.write("José\n");
    writer.write("Marta\n");
}
```

En cambio, estas clases de `java.io` no codifican bien los caracteres:

* **`FileReader` y `FileWriter`**: con ellas no tienes **ningún control sobre la codificación** de caracteres (usará la codificación establecida por el sistema operativo y la configuración de Java). Si el fichero contiene acentos o caracteres especiales, puedes encontrarte con sorpresas desagradables.

* **`RandomAccessFile`** es aún más rara, pues mezcla un montón de cosas en la misma clase: puede escribir UTF-8, pero es **un UTF-8 peculiar** que solo entiende `RandomAccessFile` (porque añade la longitud del texto antes de escribirlo). También puede leer/escribir Strings con `readLine()`/`writeBytes()`, pero en tal caso usa la codificación **UTF-16 BE**, que es una forma de Unicode extrañísima que casi nadie usa.

En definitiva, si vas a usar NIO2, codifica tus ficheros de texto siempre con UTF-8 usando `Files.newBufferedReader` y `Files.newBufferedWriter`. Y si vas a usar `java.io`, utiliza las combinaciones de clases que mencionamos antes para codificar correctamente tus caracteres con UTF-8 también.
