## 1.1. ¿Qué es JDBC?

Una de las principales aplicaciones de cualquier lenguaje moderno es la posibilidad de utilizar datos pertenecientes a un sistema de base de datos. La dificultad del manejo de archivos y las facilidades de manejo de datos que ofrecen los <abbr title="Sistemas Gestores de Base de Datos">SGBD</abbr> son los causantes de esta necesidad. 

En el mercado hay gran cantidad de bases de datos y cada una de ellas se maneja de un modo diferente. Esto está en contra del planteamiento fundamental de Java que intenta que la programación sea independiente de la plataforma. 

Hoy en día hay que tener en cuenta que la inmensa mayoría de los SGBD administran bases de datos relacionales. Éstas son bases de datos que permiten organizar los datos en tablas (relaciones) que después se interconectan mediante claves primarias y ajenas y que trabajan con el lenguaje estándar SQL. 

La idea de Java es desarrollar una sola API para el acceso unificado a bases de datos, fuera cual fuera el SGBD que las maneje. Esta interfaz se conoce como **<abbr title="Java Data Base Connect">JDBC</abbr>**. 

Las características de esta API son: 

* **JDBC es una API a nivel SQL** (el lenguaje SQL sería el que realizaría la conexión con las bases de datos), independiente de la plataforma.

* **JDBC es similar al funcionamiento de otras APIs** para acceso a bases de datos existentes (en especial a la ODBC de Microsoft, que durante muchos años fue un estándar para acceso a BDs aunque ahora se considera obsoleta). 

* **JDBC es sencilla** de manejar. 

## 1.2. ¿Qué es ODBC?

<abbr title="Open Data Base Connectivity">ODBC</abbr> es un interfaz para el manejo de datos de diversas bases de datos. Esta API de bases de datos fue muy popular hace algunos años, pero su importancia en la actualidad es cada vez más reducida. 

Su éxito se basó en la facilidad de instalación y configuración bajo sistemas Windows y en el amplio soporte para casi cualquier tipo conocido de base de datos. Pero tenía varios inconvenientes muy serios: 

* No es portable a otros los sistemas.

* Está creada en **C**, con los problemas que supone eso para la programación en otros lenguajes (punteros void por ejemplo). Esto, en definitiva, supone que si no se utiliza C para conectar con ODBC, se han de diseñar librerías intermedias de acceso a ODBC; con lo que se multiplican las capas, disminuye el rendimiento y se dificulta el desarrollo. 

* Es compleja de usar. 

Debido a su importancia decreciente en el desarrollo de software, sólo la mencionamos aquí como **curiosidad histórica**. 

## 1.3. Estructura de JDBC

En el diagrama siguiente se puede apreciar como la idea es que **las aplicaciones sólo se tengan que comunicar con el interfaz JDBC**, y en ningún caso directamente con el SGBD. Es JDBC quien se encarga de eso:

![ud09_jdbc_01.png](/img/user/attach/Unidad%2009/ud09_jdbc_01.png)

Esto hace que la programación de aplicaciones de acceso a bases de datos no necesite  conocer el funcionamiento del SGBD en particular; lo que hay que conocer es el funcionamiento de JDBC. Por supuesto, es necesario adquirir un controlador JDBC para el sistema gestor de base de datos que utilicemos. 

La comunicación fundamental entre las aplicaciones y JDBC se realiza a través de instrucciones SQL.

## 1.4. Controladores JDBC

Una vez instalado, configurado y puesto en funcionamiento nuestro sistema gestor de base de datos favorito, si queremos que las bases de datos creadas por él sean accesibles desde los programas Java, necesitamos el controlador JDBC de ese sistema. 

Hay cuatro tipos de controladores: 

* **Tipo 1. Controlador que traduce de JDBC a ODBC**. Un controlador de este tipo es la pasarela JDBC-ODBC. No es muy productiva ya que necesita ser configurada para un controlador ODBC concreto. Aunque actualmente existen mejores soluciones de este tipo, sigue requiriendo tener dos controladores en el cliente. 

   ![ud09_controladores_tipo1_jdbc_01.png](/img/user/attach/Unidad%2009/ud09_controladores_tipo1_jdbc_01.png)

* **Tipo 2. Controladores parcialmente escritos en Java y parcialmente escritos en el código nativo que comunica con el API de la base de datos**. En estos controladores hay que instalar tanto paquetes Java como paquetes de código nativo, lo que los invalida en su uso para aplicaciones web. 

   ![ud09_controladores_tipo2_jdbc_01.png](/img/user/attach/Unidad%2009/ud09_controladores_tipo2_jdbc_01.png)

* **Tipo 3. Paquetes puros de Java que usan protocolos independientes de la base de datos**. Es decir, las instrucciones JDBC son pasadas a un servidor genérico de base de datos que utiliza un protocolo determinado. No requiere tener instalado ningún software de librería de la base de datos. Lo malo es que el servidor intermedio (middleware) suele requerir instrucciones específicas de la base de datos.

   ![ud09_controladores_tipo3_jdbc_01.png](/img/user/attach/Unidad%2009/ud09_controladores_tipo3_jdbc_01.png)

* **Tipo 4. Paquetes de Java puro que traducen peticiones JDBC a protocolo de base de datos específico**. No requieren intermediarios entre el software JDBC y la base de datos. 

   ![ud09_controladores_tipo4_jdbc_01.png](/img/user/attach/Unidad%2009/ud09_controladores_tipo4_jdbc_01.png)


**El 99% de los controladores que se utilizan en la actualidad son de tipo 4**.

Para adquirir estos controladores es necesario ponerse en contacto con el fabricante o dirigirse a su página web y después descargarlo. Las instrucciones de instalación las dará el fabricante, pues pueden ser un poco diferentes según el controlador.
