Para usar JDBC, como hemos visto, necesitas una librería adicional: **el controlador JDBC de tu base de datos**. Por ejemplo, el **controlador JDBC de MySQL**.

Puedes enlazar esa librería adicional a tu proyecto de 3 maneras:

1. Descargando la librería y compilándola manualmente con el resto de nuestro código (poco recomendable en proyectos grandes)

2. Configurando el proyecto con un gestor de proyectos como Maven (difícil)

3. Dejando que sea tu IDE preferido el que configure Maven para compilar con el controlador JDBC.

La **tercera opción es la más frecuente** y la que preferiremos en este curso.  

## 2.1. Instalación manual del controlador JDBC

Si quisiéramos compilar manualmente tu proyecto con el controlador JDBC, lo primero es descargarlo.

Si el Driver es de tipo 4 (lo más habitual), podemos instalarlo en nuestro sistema manualmente si aún no lo está. Tomaremos como ejemplo MySQL. Si no está instalado, tendremos que descargarlo de [http://dev.mysql.com](http://dev.mysql.com). ¡Siempre de la web oficial del fabricante de la base de datos! 

Nos bajaremos un archivo comprimido, lo desempaquetaremos en algún sitio y nos quedaremos con el archivo **`mysql-connector-java-X.X.X-bin.jar`** (o algo parecido; las X indican el número de versión) que viene dentro. En ese jar está la clase Driver que nos interesa.

Tendremos que poner ese jar accesible en nuestro proyecto. Para eso hay varias opciones:

* Guardarlo en la carpeta donde tengas instaladas las librerías estándar de Java. Típicamente, en Windows ese directorio será `C:\Archivos de Programa\Java\jdkXXX\jre\lib\ext`. En Linux podría ser algo como `/usr/java/jdkXXX/jre/lib/ext`, pero depende de la versión de nuestro sistema. Ten en cuenta que algunos IDEs no usarán estas librerías, sino que instalarán su propia copia del JDK y del JRE de Java, por lo que esta solución podría no funcionarte con esos IDEs.

* Hacer accesible el archivo `.jar` en la variable de entorno CLASSPATH (puede que nuestro IDE  no haga caso de esa variable). Si trabajas con el símbolo del sistema, puedes teclear esto:

   ```bash
   $ set CLASSPATH=<PATH_DEL_JAR>mysql-connector-java-X.X.X-bin.jar
   ```
  
* Añadir el archivo `.jar` como librería adicional en nuestro IDE (Netbeans, Eclipse, o el  que sea). Suele hacerse en las propiedades del proyecto, pero depende mucho del IDE y del tipo de proyecto. Tendrás que bichear por Internet para aprender a hacerlo, como de costumbre.

## 2.2. Instalación del controlador JDBC en NetBeans

En cada IDE, la instalación del controlador JDBC se hará de forma diferente y resulta imposible explicar aquí todas las combinaciones posibles. Vamos a mostrar aquí solo la instalación en un **proyecto Maven con NetBeans**, que es una configuración muy simple para crear aplicaciones con entorno gráfico de usuario.

El procedimiento a seguir es el siguiente:

1. Descarga, como hemos descrito antes, el contector para MySQL de la web de MySQL. Extrae el archivo `mysql-connector-java-X.X.X.jar` en un directorio que tengas bien localizado.

2. Crea un proyecto nuevo de tipo Maven JavaFX con FXML (File → New Project → Java with Maven → FXML Java Maven)

3. Despliega los archivos del proyecto y localiza la carpeta "Dependencies"

4. Haz clic con el botón derecho en "Dependencies" y elige la opción "Add dependency"

   * En “Group id” escribe “com.mysql”

   * En “Artifact id” escribe “mysql-connector-j”

   * En “Version” escribe el número de la versión que figura en tu archivo .jar. 

   * Pulsa el botón “Add” cuando hayas acabado.

5. Despliega la carpeta “Dependencies” de tu proyecto y localiza la nueva dependencia, que debe llamarse algo así como mysql-connector-j-x.x.x.jar.

   * Haz clic con el botón derecho sobre mysql-connector-j-x.x.x.jar

   * Selecciona “Manually install dependency”

   * Localiza el archivo .jar que has descargado de la web de MySQL y selecciónalo para instalarlo localmente en el proyecto.

6. Por último, localiza la carpeta “Source packages” → “Default package” de tu proyecto:

   * Abre el archivo module-info.java

   * Añade la línea **`requires java.sql`**; debajo del resto de requerimientos.

Y listo. Con esto, tu proyecto JavaFX de NetBeans debería poder hacer un `import` de `java.sql.*` y usar todas las clases del conector JDBC sin problemas.

## 2.3. Instalación del controlador JDBC en Visual Studio Code

En **Visual Studio Code**, el proceso es bastante similar al de Netbeans, aunque tendrás que tocar un poco los archivos de configuración.

El procedimiento para usar el controlador JDBC de MySQL en VS Code es el siguiente:

1. **Crea un proyecto** Maven con Visual Studio Code. Por ejemplo, un proyecto JavaFX, que VS Code gestiona con Maven. Si no sabes cómo hacerlo, revisa el tema anterior, en el que hablamos de JavaFX.

2. **Añade al archivo pom.xml**, donde se configura el proyecto Maven, la siguiente dependencia:

   ```xml
   <dependencies>
       <dependency>
           <groupId>mysql</groupId>
           <artifactId>mysql-connector-j</artifactId> <version>8.3.0</version>
           <version>8.X.X</version>
       </dependency>
   </dependencies>
   ```

Sustituye `8.X.X` por la última versión disponible del conector JDBC para MySQL. Debe ser algo como, por ejemplo, `8.3.0`

3. **Añade al archivo module-info.java** estas dependencias:
   ```
   requires java.sql;
   requires mysql.connector.java
   ```

Con eso, el proyecto debería compilarse y ejecutarse sin problema en VS Code.

## 2.4. Conexión con el controlador

Para conseguir conectar una base de datos con una aplicación, nuestra aplicación requiere la URL de la base de datos y las propiedades que establezca nuestro controlador JDBC. Las clases necesarias para usar JDBC están en el paquete java.sql. 

En versiones más antiguas del JDBC, era necesario cargar el controlador en la aplicación Java mediante el método estático **`forName`** de la clase `Class`: 

```java
Class.forName(rutaDelDriver);
```

La ruta del driver tiene que venir en las instrucciones del fabricante. Por ejemplo en el caso del controlador MySQL el formato es: 

```java
Class.forName("com.mysql.cj.jdbc.Driver");
```

Esa instrucción puede dar lugar a las excepciones `ClassNotFoundException` (si no se encontró la clase en el drivr JDBC), `InstantiationException` (si no se puede crear el driver para la base de datos) e `IllegalAccessException` (si el acceso a la base de datos no fue correcto).

> [!warning] Importante 
> A partir de JDBC 4.0 **ya no es necesario hacer esto**, porque Java cargará automáticamente el driver si se encuentra en el CLASSPATH del sistema.

Una vez que el controlador se ha registrado, entonces se abre la URL a la base de datos, cuyo formato suele ser: 

```
jdbc:sgbd://servidor:puerto/basedatos
```

Por ejemplo, para abrir la base de datos llamada `prueba` en MySQL que tenemos en local:

```
jdbc:mysql://localhost:3306/prueba
```

La conexión se realiza mediante un objeto de la clase `java.sql.Connection`. La construcción típica implica indicar la URL de la base de datos, el usuario y la contraseña. Ejemplo (MySQL): 

```java
String cadenaConexion = "jdbc:mysql://localhost:3306/bdname";
String usuario = "root";
String password = "passwd";

Connection con = DriverManager.getConnection(
        cadenaConexion,
        usuario,
        password
);

```

El método estático `getConnection()` de la clase `DriverManager` es el encargado de realizar la conexión. Al crearla pueden ocurrir excepciones `SQLException` que habrá que capturar. 

La conexión se cierra con el método `close()` de la clase `Connection`. 

## 2.5. Ejecución de sentencias SQL

El método `createStatement()` de la clase `Connection` devuelve una variable de la interfaz Statement que nos permitirá ejecutar sentencias SQL sobre la base de datos. Los objetos Statement se crean de esta forma:

```java
Statement st = con.createStatement(); //con es un objeto Connection
```


### 2.5.1. Sentencias de definición y manipulación de datos

Para la ejecución de sentencias SQL del tipo UPDATE, INSERT o DELETE, y también CREATE TABLE , DROP TABLE y otras de definición de la estructura de las tablas, se utiliza el método **`executeUpdate()`** del interfaz `Statement`.

Este método devuelve un entero que indica el número de filas afectadas por la sentencia SQL.

Ejemplo: 

```java
String cadenaConexion = "jdbc:mysql://localhost:3306/personas";
String usuario = "root";
String password = "";

try (Connection con = DriverManager.getConnection(
         cadenaConexion, usuario, password);
     PreparedStatement st = con.prepareStatement(
         "UPDATE clientes SET sexo=? WHERE sexo=?")) {

    st.setString(1, "V");   // Valor para la primera ?
    st.setString(2, "H");   // Valor para la segunda ?


    System.out.println(st.executeUpdate());
} catch (SQLException e) {
    System.out.println(e.getMessage());
}
```

> [!note] Recuerda... 
> Los objetos que implementan la interfaz `Autocloseable` pueden instanciarse en el `try-with-resources` para que, una vez ejecutado el bloque de código, se cierren automáticamente.
> Es el caso de los objetos `Connection` y `PreparedStatement` del ejemplo anterior.

### 2.5.2. Sentencias de consulta de datos

Para lanzar sentencias SELECT a la base de datos se utiliza el método **`executeQuery()`** de **`Statement`**. Este tipo de consultas devuelven un conjunto de resultados que siempre tiene forma de tabla (temporal), y que Java representa con objetos de clase **`ResultSet`**. 

La clase `ResultSet` dispone de los típicos métodos de manipulación de cursores para procesar el conjunto de resultados. Estos métodos son sospechosamente parecidos a los métodos para recorrer archivos secuenciales.

* El método **`next`()** de `ResultSet` permite avanzar de fila. Devuelve `false` si ya no quedan filas que procesar, y true en caso contrario.

* Los métodos **`getInt("nombreCampo")`**, **`getString("nombreCampo")`**, **`getDouble("nombreCampo")`**, etc, permiten recuperar los valores de la fila actual.

Por lo tanto, una vez lanzada la consulta recorreremos el conjunto de resultados como si de un archivo secuencial binario se tratase. Es decir: se recuperan los datos de la primera tupla mediante métodos get, y luego se avanza el cursor hasta la siguiente tupla. El proceso se repite mientras queden tuplas que procesar en el conjunto de resultados.

En este ejemplo puedes ver un caso habitual de recorrido por un conjunto de resultados. Vamos a mostrar en la consola el nombre y la edad de los empleados almacenados en la tabla de la BD. Observa cómo se usa `next()` como condición del `while` que procesa todos los resultados de la consulta.

```java
String url = "jdbc:mysql://localhost:3306/personas";
String user = "root";
String pass = "";

try (Connection con = DriverManager.getConnection(url, user, pass);
        Statement st = con.createStatement();
        ResultSet rs = st.executeQuery("SELECT * FROM empleados")) {

    while (rs.next()) {
        System.out.println("Nombre: " + rs.getString("Nombre"));
        System.out.println("Edad: " + rs.getInt("Edad"));
    }
} catch (SQLException e) {
    System.out.println(e.getMessage());
}
```

Una forma alternativa muy interesante del código anterior sería esta. Observa bien cómo cambia la llamada a `executeQuery()` por **`prepareStatement()`**:

```java
import java.sql.*;

String url = "jdbc:mysql://servidor:3306/nombre-BD";
String user = "root";
String pass = "";

try (Connection con = DriverManager.getConnection(url, user, pass);
        PreparedStatement pst = con.prepareStatement(
            "SELECT * FROM usuarios WHERE nombre = ?")) {

    String n = System.console().readLine();

    pst.setString(1, n);
    try (ResultSet rs = pst.executeQuery()) {
        while (rs.next()) {
            System.out.println(rs.getString("nombre"));
            System.out.println(rs.getInt("edad"));
        }
    }
} catch (SQLException e) {
    System.out.println(e.getMessage());
}
```

En este caso, en lugar de ejecutar la instrucción inmediatamente, la preparamos antes usando interrogaciones en los lugares donde queremos incrustar los valores de nuestras variables. Posteriormente, con `setString()`, se asigna el valor necesario a la primera interrogación.

Si hubiera más interrogaciones en la instrucción SQL, serían necesarias más instrucciones `setString()`, o también `setInt()`, `setDouble()`, etc, para asignar valores a cada una de las incógnitas.

Esta forma de lanzar instrucciones SQL con preparación previa tiene varias ventajas:

1. Evita ataques por inyección SQL. La inyección SQL es un tipo de ataque muy frecuente contra las bases de datos. Aunque ahora no vamos a entrar en detalle sobre ello, debes saber que si usamos `prepareStatement()` estaremos protegiendo nuestra aplicación contra este tipo de ataques.

2. Tiene mejor rendimiento. En general, las instrucciones SQL preparadas se ejecutan más rápidamente que las que se lanzan directamente.

3. El código es más claro y legible (aunque hay que escribir un poco más)

4. El código es más fácil de depurar, puesto que no hay que abrir y cerrar interminables secuencias de comillas.

### 2.5.3. Cursores secuenciales y bidireccionales

La mayoría de los drivers JDBC permiten el uso de cursores bidireccionales, esto es, en los que el `ResultSet` puede recorrerse no solo de forma secuencial (de principio a fin), sino de forma aleatoria. En estos casos, el `ResultSet` dispone de varios métodos adicionales, como, por ejemplo:

* **`next()`**: funciona como `next()` en los cursores secuenciales: avanza a la siguiente tupla. Devuelve true si lo pudo hacer con éxito o false si ya no hay más tuplas.

* **`previous()`**: retrocede a la tupla anterior en el conjunto de resultados. También devuelve true o false.

* **`first()`**: coloca el cursor en la primera tupla.

* **`last()`**: coloca el cursor en la última tupla.

* **`beforeFirst()`**: coloca el cursor antes de la primera tupla, es decir, como si el `ResultSet` acabase de ser abierto.

* **`afterLast()`**: coloca el cursor después de la última tupla, es decir, como si hubiéramos recorrido completamente el conjunto de resultados.

* **`relative(int)`**: mueve el cursor hacia adelante el número de tuplas especificado, contando desde la posición actual del cursor. Puede ser un número negativo, en cuyo caso el desplazamiento se realiza hacia atrás.

* **`absolute(int)`**: mueve el cursor hacia adelante el nḿero de tuplas especificado, contando desde el inicio del `ResultSet`.

¡Atención! Con un `ResultSet` cualquiera solo se puede ejecutar el método `next()`. Para que el resto de métodos estén disponibles, el driver JDBC tiene que permitirlo y, además, debemos crear el `ResultSet` con un parámetro adicional en `createStatement`. Este parámetro puede valer:

* **`ResultSet.TYPE_FORDWARD_ONLY`**: creará un cursor convencional, es decir, que solo se puede recorrer secuencialmente.

* **`ResultSet.TYPE_SCROLL_INSENSITIVE`**: creará un cursor que puede moverse hacia adelante y hacia atrás.

* **`ResultSet.TYPE_SCROLL_SENSITIVE`**: como el anterior, pero si los datos de la BD cambian durante el procesamiento de `ResultSet`, se actualizarán también el el programa Java.

Por ejemplo, la siguiente sentencia SELECT se recuperará en un `ResultSet` bidireccional:

```java
Statement stmt = con.createStatement(ResultSet.TYPE_SCROLL_INSENSITIVE);  
ResultSet rs = stmt.executeQuery("SELECT a, b FROM miTabla");
```

### 2.5.4. Aspectos avanzados: concurrencia y persistencia

Existen otras variantes de ResultSet que permiten construir cursores ligeramente distintos. Por ejemplo, algunos drivers JDBC permiten utilizar cursores que no solo pueden recuperar los datos de la BD, sino también modificarlos. Es decir, que si cambiamos un dato residente en un ResultSet, automáticamente el driver si encarga de cambiar ese mismo dato en la BD.

También los ResultSet pueden hacerse resistentes a los problemas derivados de la persistencia de los datos. Si realizamos una consulta y la almacenamos en un ResultSet para procesarla posteriormente en nuestro programa Java, y otro usuario realiza un `commit` en la BD que afecte a los datos que hemos recuperado, nuestro programa podría fallar. Al crear el cursor podemos establecer ciertos parámetros para que éste reaccione adecuadamente ante esta posibilidad.

Ambas cuestiones (concurrencia y persistencia) son fuertemente dependientes del driver JDBC y del SGBD que estemos usando, y exceden a los propósitos de este curso introductorio de programación.

### 2.5.5. Correspondencia de tipos entre Java y SQL

Para utilizar correctamente los métodos `getInt()`, `getString()`, `getDouble()`, etc, de un `ResultSet`, es necesario conocer la correspondencia entre los tipos Java y los tipos SQL.

| Tipo SQL | Tipo JDBC (java.sql.Types) | Clase Java (Tradicional) | Clase Java (Recomendada en la actualidad) |
| ----- | ----- | ----- | ----- |
| `CHAR, VARCHAR` | `VARCHAR, CHAR` | `String` | `String` |
| `BIT, BOOLEAN` | `BIT, BOOLEAN` | `boolean` | `boolean` |
| `TINYINT` | `TINYINT` | `byte` | `byte` |
| `SMALLINT` | `SMALLINT` | `short` | `short` |
| `INTEGER` | `INTEGER` | `int` | `int` |
| `BIGINT` | `BIGINT` | `long` | `long` |
| `REAL, FLOAT` | `REAL, FLOAT` | `float` | `float` |
| `DOUBLE` | `DOUBLE` | `double` | `double` |
| `DECIMAL, NUMERIC` | `DECIMAL, NUMERIC` | `java.math.BigDecimal` | `java.math.BigDecimal` |
| `DATE` | `DATE` | `java.sql.Date` | `java.time.LocalDate` |
| `TIME` | `TIME` | `java.sql.Time` | `java.time.LocalTime` |
| `TIMESTAMP` | `TIMESTAMP` | `java.sql.Timestamp` | `java.time.LocalDateTime` |
| `BLOB` | `BLOB` | `java.sql.Blob` | `byte[] o InputStream` |
| `CLOB` | `CLOB` | `java.sql.Clob` | `String o Reader` |


### 2.5.6. Valores nulos

Al leer un determinado campo con alguna función get puede ocurrir que el valor leído sea nulo. En ese caso se puede comprobar mediante el método **`wasNull()`**. Este método devuelve true si la última llamada a get obtuvo un valor nulo de la base de datos. 

Ejemplo: 

```java
int n = rs.getInt("Valor");   
if (rs.wasNull()){   
    System.out.println("Se leyó un valor nulo");   
} 
```
