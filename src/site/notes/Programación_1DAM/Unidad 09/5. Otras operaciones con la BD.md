## 5.1. Transacciones

Las transacciones, como sabrás por el módulo de Bases de Datos, son secuencias de operaciones SQL que deben ejecutarse en bloque. Es decir, o se ejecutan todas, o no se ejecuta ninguna, pues la ejecución parcial dejaría a las tablas en un estado inconsistente.

Todas las instrucciones de inserción, actualización o borrado que se ejecuten desde Java se lanzan, por defecto, en ese mismo instante, porque está habilitado el modo auto-commit. Para deshabilitarlo y poder realizar transacciones complejas hay que ejecutar:

```java
// con es el objeto conexión
con.setAutoCommit(false);
```

A partir de ese momento, podemos ejecutar el método con.rollback() si algún paso de la transacción falla, o volver a hacer con.setAutoCommit(true) si queremos finalizar la transacción con éxito.

Por ejemplo:

```java
try {
    con.setAutoCommit(false);

    try (Statement st = con.createStatement()) {
        // Aquí vienen las sentencias SQL de la transacción
        st.executeUpdate("Sentencia SQL 1");
        st.executeUpdate("Sentencia SQL 2");
        // ...
        st.executeUpdate("Sentencia SQL N");
    }

    con.commit();
} catch (SQLException e) {
    if (condicionParaHacerRollback) {
        try {
            con.rollback();
        } catch (SQLException rollbackEx) {
            rollbackEx.printStackTrace();
        }
    }
    e.printStackTrace();
} finally {
    try {
        con.setAutoCommit(true);
    } catch (SQLException ex) {
        ex.printStackTrace();
    }
}
```

## 5.2. Funciones de usuario

En SQL es habitual utilizar funciones como `COUNT(*)`, `MAX(id)`, `MIN(columna)`, etc, que proporcionan resultados calculados a partir de los datos de la consulta.

Los SGBD permiten crear funciones personalizadas para complementar a las que ya existen en SQL. Estas funciones almacenadas son parecidas a los procedimientos almacenados, y se caracterizan porque:

* Siempre devuelven un valor (los procedimientos, no)

* El valor devuelto es un tipo de datos simple, nunca un `ResultSet`.

* Pueden utilizarse en un `SELECT.`

* Admiten parámetros de entrada.

Desde Java pueden utilizarse tanto las funciones SQL estándar como las definidas por el usuario. Será necesario, en este caso, acceder al campo correspondiente mediante su número de orden en la consulta, y no mediante su nombre. Por ejemplo:

```java
Statement st = con.createStatement();  
ResultSet rs = st.executeQuery("SELECT COUNT(id) FROM usuarios");  
rs.next();  
System.out.println("Hay " + rs.getInt(1) + " usuarios en la BD");  
// Intentar rs.getInt("COUNT(id)") fallaría
```

## 5.3. Procedimientos almacenados

También has debido ver en el módulo de Bases de Datos cómo muchos SGBD permiten crear procedimientos almacenados dentro de la propia BD, que sirven para estandarizar y automatizar ciertas tareas complejas, normalmente asociadas a la administración y el mantenimiento de la BD.

Los procedimientos son más flexibles que las funciones, porque pueden devolver 0, 1 o varios datos. Pero tienen una limitación: no se pueden ejecutar desde una consulta, sino que deben ser invocados explícitamente. Por ejemplo, desde una aplicación externa escrita en Java.

Los procedimientos almacenados pueden tener parámetros de entrada (IN), de salida (OUT) o de entrada-salida (INOUT). Para usarlos desde Java tenemos que:

1. Crear un objeto de la clase `CallableStatement`, en lugar de `Statement`. Utilizaremos para ello el método `prepareCall()` de la conexión.

2. Indicar a Java el número y tipo de los parámetros del procedimiento.

3. Ejecutar el procedimiento.

Vamos a verlo con un ejemplo. Supongamos un procedimiento almacenado como este:  

```sql
CREATE PROCEDURE cuentaUsuarios(OUT c INT)  
BEGIN  
    SELECT COUNT(\*) INTO c FROM usuarios;  
END
```

El procedimiento, como sin duda habrás averiguado, devuelve en el parámetro de salida c el número de registros que hay en la tabla usuarios.

La forma de invocar este procedimiento desde Java y recuperar el resultado devuelto en el parámetro `c` sería la siguiente:

```java
try {
   CallableStatement cs = con.prepareCall("{call cuentaUsuarios(?)}");
   cs.registrerOutParameter(1, Types.INTEGER);
   cs.execute();
   int cuenta = cs.getInt(1);
   System.out.println("El número de usuarios es " + cuenta);
}
   catch (SQLException e) {e.printStackTrace(); }
}
```

En la primera línea creamos el objeto de tipo `CallableStatement` mediante el método `prepareCall()` de la conexión. Ahí indicamos el nombre del procedimiento almacenado (`cuentaUsuarios`) y el número de parámetros que tiene (uno en este caso), indicados como interrogaciones.

La siguiente línea indica a Java el tipo de los parámetros: el parámetro 1 es de tipo Integer y de salida (por eso se emplea `registerOutParameter()`)

Por último, el método `execute()` provoca la ejecución del procedimiento almacenado.

Más abajo se recoge el valor devuelto en el parámetro de salida c mediante `getInt()`, a la que se le pasa el número del parámetro que queremos recuperar. Como es lógico, este parámetro debe ser de salida o de entrada/salida. Si es un parámetro de entrada, `getInt()` fallará.

Asimismo, existen otros métodos `getDouble()`, `getString()`, `getByte()`, etc, para recuperar los valores de otros tipos de parámetros.

En el caso de tener parámetros de entrada, podemos establecer su valor antes de ejecutar el procedimiento con los método análogos `setInt()`, `setString()`, `setDouble()`, etc. 

Veámoslo con otro ejemplo. Supongamos que ahora disponemos de un procedimiento almacenado de este tipo:

```sql
CREATE PROCEDURE miProcedimiento(IN a VARCHAR(100), OUT b INT, INOUT c INT)  
...
```

Para usarlo desde Java haríamos algo como esto:

```java
try {
   CallableStatement cs = con.prepareCall("{call miProcedimiento(?,?,?)}");
   cs.setString(1, "Hola mundo");
   cs.registrerOutParameter(2, Types.INTEGER);
   cs.setInt(3, 0);
   cs.registrerOutParameter(3, Types.INTEGER);
   cs.execute();
   int b = cs.getInt(2);
   int c = cs.getInt(3);
}
   catch (SQLException e) {e.printStackTrace(); }
}
```

Observa como el parámetro 1, que es de entrada, no necesita ser registrado, pero sí que hay que asignarle un valor con `setString()`. El parámetro 3, al ser de entrada/salida, necesita ambas cosas: registrarse con `registerOutParameter()` y recibir un valor con `setInt()`.
