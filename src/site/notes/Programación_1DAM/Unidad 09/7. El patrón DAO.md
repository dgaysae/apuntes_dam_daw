El código de la sección anterior mezcla el manejo del interfaz gráfico con el acceso a la base de datos, puesto que las dos cosas se han hecho desde el controlador.

**Esto se considera una mala práctica**. A medida que las aplicaciones crecen, escribir el código JDBC (conexiones, `try-catch`, SQL) directamente dentro de los controladores se vuelve inmanejable. Para solucionar esto, se ha creado el **Patrón <abbr title="Data Access Object">DAO</abbr>**. 



## 7.1. ¿Qué es el patrón DAO?

El patrón DAO es un **patrón de arquitectura**, es decir, una forma predefinida de organizar el código de clases que se ha visto que se adapta a muchos proyectos con éxito.

El objetivo del patrón DAO es **aislar la lógica de negocio de la lógica de acceso a datos**. De esta forma, si el día de mañana cambiamos de base de datos (por ejemplo, de MySQL a MongoDB), solo tendremos que modificar una de las clases del programa, mientras que el resto de la aplicación permanecerá intacta. Del mismo modo, si queremos cambiar el interfaz gráfico por un interfaz web, solo habría que cambiar una de las capas del programa, y el resto seguiría igual.

## 7.2. Componentes de DAO

Para implementar este patrón, solemos crear cinco elementos:

1. **Clase de Modelo**: Es una clase Java simple que representa una tabla de la base de datos (p. ej. Alumno.java, Cliente.java, Producto.java, o lo que corresponda con nuestra base de datos). Los atributos de esa clase deben coincidir en nombre y tipo con los campos de la tabla a la que representa.

2. **Interfaz DAO** (optativo pero recomendable): Define qué operaciones se pueden hacer, pero no cómo se hacen.

3. **Implementación DAO**: Contiene el código JDBC (es decir, las instrucciones SQL). Utiliza objetos de tipo Modelo para instanciar registros de la tabla y pasárselos al interfaz gráfico o recibirlos de él. Si se ha usado un interfaz DAO, esta clase debe implementar ese interfaz.

4. **Clase Conexión**: Una clase encargada de gestionar la conexión a la base de datos de forma centralizada, de manera que solo se pueda abrir una conexión simultáneamente.

5. **Interfaz de usuario**: El interfaz de usuario se maneja en otra clase diferente, que accederá a la implementación DAO para obtener los datos de la BD o para enviar datos a la BD. Si usamos JavaFX, este interfaz de usuario lo constituirán los controladores y vistas. Si usamos un interfaz de texto, será una clase que centralizará la entrada/salida por teclado y pantalla.

## 7.3. Un ejemplo de implementación

Imaginemos que tenemos una tabla de clientes con los campos id, nombre y email. 

### 1) Clase modelo (`Cliente.java`)
Para manejar la tabla clientes según el patrón DAO, deberíamos crear una **clase `Cliente`** que funcione como **modelo de datos** de esa tabla. Cada objeto `Cliente` se corresponde con un registro de la tabla:

```java
// Modelo Cliente (Cliente.java)
public class Cliente {
    // Aquí se ponen todos los campos que tenga la tabla de clientes
    private int id;
    private String nombre;
    private String email;

    // Añadir constructores, Getters y Setters
    // Y también el equals/hashCode y la implementación de Comparable.
}
```

### 2) Interfaz DAO (`DAO.java`) - Optativo pero recomendable  

Opcionalmente, se puede declarar un interfaz con las operaciones que se quieren realizar sobre la tabla `Cliente` o cualquier otra de la aplicación. Por ejemplo, el siguiente interfaz usa un tipo genérico `T` para que sirva para `Cliente` o para cualquier otro modelo, y define varias operaciones típicas con la base de datos, como `findAll()`, `insert()` o `delete()`. 

```java
public interface DAO<T> {  
    List<T> findAll() throws SQLException;  
    T findById(int id) throws SQLException;  
    void insert(T entidad) throws SQLException;  
    void update(T entidad) throws SQLException;  
    void delete(int id) throws SQLException;  
}
```

### 3) Implementación DAO para `Cliente` (`ClienteDAO.java`)

La **clase `ClienteDAO`** funcionará como **intermediario entre el modelo `Cliente` y el interfaz de usuario**. 

Esta clase implementará el acceso a la base de datos que tengan que ver con clientes: búsquedas, inserciones, borrados y modificaciones de clientes. 

Al conjunto de operaciones típicas que suelen implementar estas clases se le denomina **`CRUD`** (Create, Read, Update, Delete).

Por ejemplo, en el siguiente ClienteDAO vamos a implementar solo el método `findAll()`  del interfaz DAO. Este método devuelve un array con todos los clientes de la tabla para que el interfaz de usuario haga con el array lo que estime oportuno (por ejemplo, mostrarlo en un `TableView`).

```java
public class ClienteDAO implements DAO<Cliente> {
    private final String URL = "jdbc:mysql://localhost:3306/mi_bd";
    private final String USER = "root";
    private final String PASS = "";

    @Override
    public List<Cliente> findAll() throws SQLException {
        List<Cliente> lista = new ArrayList<>();
        String sql = "SELECT * FROM clientes";

        try (Connection con = DriverManager.getConnection(URL, USER, PASS);
             PreparedStatement ps = con.prepareStatement(sql);
             ResultSet rs = ps.executeQuery()) {

            while (rs.next()) {
                Cliente c = new Cliente();
                c.setId(rs.getInt("id"));
                c.setNombre(rs.getString("nombre"));
                c.setEmail(rs.getString("email"));
                lista.add(c);
            }
        }
        return lista;
    }

    // Aquí añadiríamos el resto de métodos, como findById(), update(),
    // insert(), delete(), etc.
}
```

### 4) La clase Conexión (`Conexion.java`)

El último elemento típico del patrón `DAO` es la clase Conexión, que evita tener que repetir el código para conectarnos/desconectarnos de la BD en todos los modelos `DAO`. Suele manejarse con variables estáticas para evitar abrir más de una conexión simultáneamente.

Podría ser algo así:

```java
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class ConexionBD {
    private static final String URL = "jdbc:mysql://localhost:3306/mi_bd";
    private static final String USER = "root";
    private static final String PASS = "";
    
    private static Connection conexion = null;

    // Constructor privado para evitar que se creen instancias con 'new'
    private ConexionBD() {}

    public static Connection getConexion() throws SQLException {
        // Si la conexión no existe o se ha cerrado, la creamos
        if (conexion == null || conexion.isClosed()) {
            try {
                conexion = DriverManager.getConnection(URL, USER, PASS);
            } catch (SQLException e) {
                System.err.println("Error al conectar: " + e.getMessage());
                throw e; 
            }
        }
        return conexion;
    }

    public static void cerrarConexion() {
        if (conexion != null) {
            try {
                conexion.close();
            } catch (SQLException e) {
                System.err.println("Error al cerrar: " + e.getMessage());
            }
        }
    }
}
```

De este modo, cualquier modelo DAO que quiera establecer una conexión con la BD, puede hacerlo así:

```java
try (Connection con = ConexionBD.getConexion();...) {  
...  
}
```

## 7.4. Ventajas de usar el patrón DAO

Usar el patrón DAO puede parecer un poco engorroso al principio, sobre todo si el programa es pequeño: puedes tener la sensación de que lo complica todo en lugar de facilitarte la labor.

Pero, a la larga, y sobre todo cuando el tamaño del proyecto crece mucho, este patrón aporta muchas ventajas al proyecto:

* **Código más limpio**: El controlador de JavaFX (o el interfaz de usuario que estés usando) solo tiene que llamar a  `dao.findAll()`, o al método DAO que necesite, y recibirá los datos. El interfaz de usuario no ve ni una sola línea de SQL.

* **Reutilización**: Puedes usar el mismo DAO para una aplicación gráfica de escritorio (con Swing o JavaFX), para una aplicación en modo texto o para una aplicación web.

* **Mantenimiento más fácil**: Es mucho más fácil probar el código y depurarlo si está organizado en clases con responsabilidades bien separadas.

  Para terminar, una “**regla de oro**” cuando usas el patrón DAO:

> [!warning] IMPORTANTÍSIMO 
> **¡Nunca pongas un `System.out.println` o un `Alert` de JavaFX o de Swing dentro de una clase DAO o de un modelo!**
> 
> El DAO **solo debe ocuparse de los datos y lanzar excepciones** (`throws SQLException`) para que la clase donde está el interfaz de usuario decida cómo informar al usuario.
